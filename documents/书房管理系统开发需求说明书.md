# 书房管理系统开发需求说明书

## 1. 项目背景与目标

开发一套基于数据库“三级模式”（外模式 / 中间模式 / 内模式）架构的书房管理系统，通过 NFC 技术连接实体书架与数字信息，实现以下目标：

- 将实体书架位置与数字书架容器建立稳定映射。
- 支持 NFC 标签写入、读取与解析，完成物理入口交互。
- 基于逻辑书架快速查询图书数据并进行可视化展示。
- 集成豆瓣数据源，自动补全图书元数据（封面、作者、简介等）。
- 保障系统可维护性、可扩展性与数据一致性。

---

## 2. 系统范围

### 2.1 In Scope

- NFC 标签读写与实体书架映射。
- 三级模式分层实现与标准接口定义。
- 图书元数据存储、索引与查询。
- 豆瓣 爬虫（带cookie）数据拉取与更新。
- 前端逻辑书架视图与图书展示。

### 2.2 Out of Scope

- 多用户权限体系（管理员/访客角色细粒度权限）。
- 借阅、归还、罚款等图书馆业务流程。
- 大规模分布式部署与多租户隔离。

---

## 3. 三级模式架构设计

## 3.1 外模式（用户交互与 NFC 映射层）

### 功能定义

负责物理世界与数字世界的入口交互。

### 核心任务

- NFC 标签与实体书架的一一映射配置。
- 写入功能：将实体书架位置编码写入 NFC 标签。
- 读取功能：读取并解析 NFC 标签中的位置编码。

### 输入与输出

- 输入：用户选择的实体书架信息、NFC 标签 UID。
- 输出：标准化外部数据对象 `ExternalShelfBinding`。

### 异常处理

- 标签不可写/读取失败。
- 标签内容格式错误或编码损坏。
- 标签未绑定逻辑书架。

## 3.2 中间模式（逻辑映射与接口层）

### 功能定义

作为系统核心枢纽，负责数据转换与逻辑关联。

### 核心任务

- 接口 A：`NFC数据 -> 逻辑书架数据`。
- 接口 B：`逻辑书架数据 -> 内部存储数据`。

### 关键能力

- 位置编码标准化与去重校验。
- 逻辑书架容器解析与版本化映射。
- 对前端输出统一 DTO，屏蔽内存储细节。

## 3.3 内模式（底层存储与数据获取层）

### 功能定义

负责图书元数据持久化存储与外部数据源集成。

### 核心任务

- 管理图书详细数据存储。
- 集成豆瓣 API（或替代数据源）自动获取图书元数据。
- 维护底层数据与逻辑书架的索引关系。

### 数据一致性要求

- 保证 ISBN 唯一约束（可配置为同 ISBN 多副本）。
- 保证逻辑书架与图书关系完整性（外键约束或同等机制）。
- 更新失败时支持重试和补偿。

---

## 4. 数据流向与逻辑闭环

系统必须遵循以下路径：

1. 物理层：NFC 标签存储实体书架位置 ID。
2. 逻辑层：系统读取 ID，通过中间模式映射到逻辑书架容器。
3. 数据层：基于逻辑书架索引，从内模式数据库查询图书元数据（来源豆瓣）。
4. 展示层：前端将实体位置与图书元数据合并渲染。

---

## 5. 核心数据模型（建议）

## 5.1 实体：PhysicalShelf

- `physical_shelf_id` (PK)
- `location_code`（唯一，如 `study-left-3`）
- `location_name`（如“书房-左侧-第3层”）
- `nfc_tag_uid`（可空，绑定后唯一）
- `created_at`, `updated_at`

## 5.2 实体：LogicalShelf

- `logical_shelf_id` (PK)
- `shelf_name`
- `description`
- `created_at`, `updated_at`

## 5.3 关系：PhysicalLogicalMapping

- `mapping_id` (PK)
- `physical_shelf_id` (FK)
- `logical_shelf_id` (FK)
- `mapping_type`（`one_to_one` / `one_to_many`）
- `is_active`
- `version`

## 5.4 实体：BookMetadata

- `book_id` (PK)
- `isbn`（索引）
- `title`
- `author`
- `publisher`
- `publish_date`
- `cover_url`
- `summary`
- `source`（`douban` / `manual`）
- `last_sync_at`

## 5.5 关系：LogicalShelfBook

- `id` (PK)
- `logical_shelf_id` (FK)
- `book_id` (FK)
- `sort_order`
- `status`（`in_shelf` / `removed`）

---

## 6. 接口定义（建议）

## 6.1 外模式接口（NFC）

- `POST /api/nfc/write`

  - 入参：`locationCode`, `locationName`, `tagUid`
  - 出参：写入结果、校验信息
- `GET /api/nfc/read?tagUid=...`

  - 出参：`locationCode`, `locationName`, `rawPayload`

## 6.2 中间模式接口

- `POST /api/mapping/resolve`

  - 入参：`locationCode` 或 NFC payload
  - 出参：`logicalShelfId`, `logicalShelfName`, `physicalLocation`
- `GET /api/shelves/{logicalShelfId}/books`

  - 出参：书架信息 + 图书列表

## 6.3 内模式接口

- `POST /api/books/sync`

  - 入参：`isbn` 或 `keyword`
  - 处理：调用豆瓣 API 拉取并 upsert
  - 出参：同步状态、图书元数据
- `POST /api/shelves/{logicalShelfId}/books`

  - 入参：`bookId`, `sortOrder`
  - 出参：关联结果

## 7. 前端展示与交互需求

### 7.1 页面核心

- 逻辑书架视图为主。
- 需同时展示：
  - 实体信息（位置名、位置编码、NFC 标签状态）。
  - 图书详情（封面、书名、作者、简介摘要）。

### 7.2 关键交互

- 扫描 NFC 后自动跳转对应逻辑书架。
- 支持网格/列表切换。
- 支持按作者、标题关键词检索。
- 支持元数据缺失提示与“一键同步”。

### 7.3 UI 状态

- 空状态：无图书时给出引导。
- 异常状态：NFC 读取失败、映射缺失、API 超时。
- 加载状态：骨架屏或进度提示。

---

## 8. 非功能需求

- 可观测性：
  - 记录 NFC 读写日志、映射日志、数据同步日志。

---

## 9. 业务规则

- 一个 NFC 标签同一时刻仅绑定一个实体书架。
- 一个实体书架至少可映射一个逻辑书架。
- 逻辑书架可容纳多本图书。
- 图书可按业务需求允许跨逻辑书架复用（可配置）。
- 豆瓣拉取失败时允许手工补录，并标记数据来源。

---

## 10. 验收标准（UAT）

1. 能成功完成 NFC 标签写入、读取和内容校验。
2. 扫描任一已绑定标签，可在 2 秒内展示对应逻辑书架页面。
3. 逻辑书架页面可正确展示实体位置信息和图书列表。
4. 输入 ISBN 可自动拉取豆瓣元数据并入库。
5. 映射关系调整后，前端展示结果实时生效或在刷新后生效。
6. 出现读取失败、映射缺失、豆瓣异常时，系统给出明确错误提示且不崩溃。

---

## 11. 实施阶段建议

1. 第 1 阶段：完成数据模型、映射表、基础 API。
2. 第 2 阶段：接入 NFC 读写器，打通外模式 -> 中间模式链路。
3. 第 3 阶段：接入豆瓣数据同步，完成内模式数据补全。
4. 第 4 阶段：实现前端逻辑书架视图与关键交互。
5. 第 5 阶段：联调、压测、异常演练与验收。

---

## 12. 风险与应对

- 豆瓣 API 访问限制或策略变化：

  - 方案：增加缓存层、限流、重试与可替换数据源适配器。
- NFC 设备兼容性问题：

  - 方案：定义统一读写协议，增加设备抽象层与模拟器测试。
- 映射数据不一致：

  - 方案：增加映射版本号、审计日志与回滚机制。

---

## 13. 术语说明

- 外模式：面向用户与设备交互的数据视图层。
- 中间模式：业务逻辑与映射转换层。
- 内模式：数据存储与索引实现层。
- 逻辑书架：系统内聚合图书的业务容器，不等同于实体层板。
- 实体书架：物理空间中的真实位置对象。
